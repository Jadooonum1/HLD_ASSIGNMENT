Parth's Autocomplete Engine & Analytics Dashboard

A fully functioning, high-performance distributed search typeahead and suggestion backend built for the HLD (High-Level Design) assignment. It provides instant prefix completions, tracks user query trends, handles database load via aggregating write buffers, shards cached keys using consistent hashing virtual nodes, and ranks suggestions dynamically using a blended popularity and recency scorer.

3. Getting Started

Prerequisites

  • Node.js (v18+ recommended)

Quick Start Commands

  1. Install required packages:
    npm install
  2. Build the Zipfian mock dataset (120,000 queries):
    npm run generate-data
  3. Start the application:
    npm start

Once started, visit http://localhost:3000 to access the dashboard.

4. Diagnostic & Load Testing Reports

The following performance summary was generated by running the benchmark script (executing 20,000 suggest reads at 50-way concurrency, followed by 5,000 simulated search writes):

  • Server suggestion compute time (p95): 0.003 ms (Sub-millisecond response for autocomplete candidates)
  • Client-side end-to-end latency (p95): 4.88 ms (Low latency under high concurrency)
  • Cache Hit Efficiency: 99.8% (Cache Shards absorb 19,966 / 20,000 read requests)
  • Database Disk Reads: 34 (Direct lookups limited to initial cache misses)
  • Database Write Reduction: 99.3% (Consolidated 5,000 updates into 34 atomic disk commits)

5. API Reference

  • GET /suggest?q=<prefix>&mode=base|recency
    Returns top 10 ranked candidates matching the prefix.
  • POST /search (Payload: { "query": "string" })
    Records search, updates trend tracker, and buffers count increment.
  • GET /trending
    Lists current queries ranked by decayed recency.
  • GET /cache/debug?prefix=<prefix>
    Diagnostic info showing ring position hash and owner shard.
  • GET /metrics
    System counters and latency summary statistics.

  • src/storeManager.js: Map query storage and index synchronization manager
  • src/trieIndex.js: O(L) prefix candidate lookup index
  • src/querySuggester.js: Suggestion search workflow (cache check -> fallback -> cache write)
  • src/rankingScorer.js: Basic count ranking and blended popularity-recency calculations
  • src/trendTracker.js: Real-time exponential decay score tracker
  • src/hashRing.js: Consistent hashing mapping algorithm using virtual nodes
  • src/cacheManager.js: Shards prefix routes to matching cache partitions
  • src/cacheShard.js: Key-value cache shard with independent lazy TTL and FIFO eviction
  • src/statsTracker.js: Tracks rolling metrics and calculates latency percentiles
  • src/appLogger.js: Console logging wrapper
  • public/: Dark-themed glassmorphism interface (HTML, CSS, JS)

7. Deep Dive: Key HLD Decisions

  • Consistent Hash Ring: Distributes search prefixes across 4 cache shards. The use of virtual nodes (150 per node) ensures uniform key storage distribution. Node removals/additions only trigger re-allocations of 1/N keys rather than a global stampede.
  • Write Buffering: To protect the storage layer, search counts are aggregated in an in-memory buffer and flushed either when size limits are reached or every 2 seconds. The risk of losing minor updates during a crash is accepted in favor of a 99% reduction in disk write QPS.
  • Blended Recency Metric: Counts are scaled using log10(count) so older, massive query volumes do not drown out new spikes. Spikes are governed by an exponential half-life decay, ensuring short-lived spikes fade cleanly as search volumes cool down.
